* Introduction
The advent of code is (as the name suggests) an advent calendar of
sorts.  The difference between the advent of codes calendar and the
more familiar version is that the advent of code makes programming
challenges available every day instead of the usual treats (although
of course it is a treat to solve the questions ;-) ).

The challenge was (to the best of my knowledge) first offered in 2015
and although I did hear about it at the time I was too busy to take
part.  This year I was also very busy; so I manufactured the time to
do it (!) :P

* Motivation for Using Emacs Lisp
I'm a huge fan of Emacs and I've been working on and off on helping
out on some open source projects (most notably CEDET).  Aside from
minor tweaks and the work I constantly do to my own "configuration"
(read never ending program) I haven't had all that much experience in
Emacs Lisp.  So I decided to dive in and solve all these problems to:
 1. see whether E-Lisp is as slow as some people claim it is;
 2. learn more about programming in E-Lisp;
 3. have fun (!)

* Findings
Emacs Lisp is now one of my favourite languages.  It was incredibly
fun to develop the solutions to these problems in E-Lisp.  E-Lisp
feels expressive and open and doesn't require for an incredible amount
of boiler plate code to be written just to accomplish the simplest of
tasks.

** The Language
It's my opinion that many of the aspects of the language which I've
heard people complain about act to it's advantage in making it a great
language to use.  For example, the lack of name spaces and the use of
package prefixes in their wake leads to consistent semantics.  i.e. if
you include a file, it reads the lisp and evaluates it.  There is no
magic happening, and no need to read a manual to understand the
details.  It makes for a great hacking environment in which we can
freely redefine anything at will and not have to run some lengthy
process involving external tools to resolve dependencies and re import
symbols -- just evaluate the new =defun=/=defvar=/=defmacro= etc. and
it'll be there!

** The Libraries
With the CL and Seq libraries there's hardly ever a need to import
extra libraries.  On one or two occasions I used the libraries defined
by Dr. Cubit for his package "Predictive" including the heap, queue
and trie (although some of those solutions never made it into this
repo.).  Additionally I used dash for one or two of the problems.
These libraries provide a wealth of functions with higher order
interfaces which I've never imagined (such as mapping over two
sequences instead of first zipping them).

** The Performance
I only ever ran into performance problems when:
 a) I had come up with an ill conceived solution;
 b) the problem was just computationally difficult;
 c) memory was the problem;

Some of the problems are just difficult to get running fast (< 1s) and
I might not have been aware of the algorithm which would do it.
Sometimes I just implemented a solution which "got the job done"
rather than researching better means of implementing the algorithm.

When memory is the problem it was the fairly simplistic nature of the
garbage collection in Emacs did hurt performance (or so I thought
based on the profiles I generated).

All in all I was impressed by the performance of E-Lisp and although I
gauged (based on feedback from friends who were also doing the
challenge) that their languages exhibited better performance, I didn't
feel that it was so much better that it really mattered.
